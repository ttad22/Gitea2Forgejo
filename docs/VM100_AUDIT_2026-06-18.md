# VM100 Audit — 2026-06-18

This document records the read-only audit of the live TTT Gitea deployment on
`vm100` (`git-ops`) to support a same-VM Forgejo migration plan.

## Summary

The deployment is a good candidate for a staged same-VM migration because it is
small, has ample free disk, uses PostgreSQL locally, and is still on
`Gitea 1.22.0`, which remains within the last officially supported Gitea cohort
for direct Forgejo upgrade staging.

## Deployment Topology

- VM ID: `100`
- VM name: `git-ops`
- Guest address: `10.200.40.100/24`
- Guest OS access: QEMU guest agent enabled, system reachable
- On-boot: enabled

## Application Layout

- Service manager: `systemd`
- Service unit:
  - `/etc/systemd/system/gitea.service`
- Start command:
  - `/usr/local/bin/gitea web --config /etc/gitea/app.ini`
- Reverse proxy:
  - `nginx` active on `:80`
- Gitea process:
  - active on `:3000`
- SSH:
  - host `sshd` on `:22`
- PostgreSQL:
  - local listener on `127.0.0.1:5432`

## Application Version

- Gitea: `1.22.0`
- PostgreSQL: `14.23`

## Data Locations

- Config: `/etc/gitea/app.ini`
- Work dir: `/var/lib/gitea`
- App data: `/var/lib/gitea/data`
- Repositories: `/var/lib/gitea/data/gitea-repositories`
- LFS: `/var/lib/gitea/lfs`
- Attachments: `/var/lib/gitea/data/attachments`
- Logs: `/var/lib/gitea/log`

## Storage Footprint

- Root filesystem free space: about `188 GB`
- Total Gitea work directory: about `876 MB`
- Repository storage: about `795 MB`
- Attachments: about `80 MB`
- LFS storage: effectively unused
- Packages storage: effectively unused

This is a small instance by migration standards. Data size will not be the main
source of downtime. Service stop/start, package replacement, and post-upgrade
validation will dominate the window.

## Feature Use Snapshot

- Repositories: `36`
- Users: `5`
- Org memberships: `2`
- LFS DB records: `0`
- Actions runs: `0`
- Action runners: `0`
- Packages: `0`

This is favorable for migration because the major Forgejo/Gitea divergence areas
called out upstream, especially Actions and package behaviors, are not in active
use here.

## Configuration Observations

- Database backend is PostgreSQL, not SQLite.
- Internal SSH server is disabled in Gitea:
  - `START_SSH_SERVER = false`
- LFS server is enabled:
  - `LFS_START_SERVER = true`
- Registration is disabled.
- OpenID sign-in/sign-up are disabled.
- Gitea is exposed through nginx, not directly on `:80`.

## Migration Significance

This combination strongly favors a staged same-VM migration:

- same database backend before and after migration
- same filesystem layout can be preserved
- same reverse proxy tier can be preserved
- no active Actions or package workload to translate
- little to no LFS migration burden
- enough free disk for local backups and rollback artifacts

## Risk Notes

1. The migration should not be treated as a direct jump to the newest Forgejo.
   The supported path is staged.
2. Secrets in `app.ini` must be preserved intact.
3. nginx and clone URLs must remain stable during cutover.
4. Any custom assets under the old Gitea layout should be checked and moved into
   `custom/` if needed by Forgejo expectations.
